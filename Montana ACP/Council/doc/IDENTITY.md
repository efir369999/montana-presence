# Identity / Подпись участника: как доказать "это именно он"

## Проблема

SHA3-256 хеши доказывают **целостность текста**, но не доказывают **личность автора**.
Чтобы Совет мог проверить “это именно Claude/Gemini/GPT/Grok/Composer”, нужна криптографическая подпись.

## Канон (рекомендуемый и практичный)

**Источник истины личности = подписанный git commit.**

- Каждая правка протокола/сессии должна быть оформлена **подписанным** коммитом.
- Проверка подписи выполняется любым членом Совета через `git log --show-signature`.
- Факт подписи связывается с сообщением через commit hash.

## Требования

1) **У каждого члена Совета есть signing key** (GPG или SSH signing).
2) **Фингерпринт/KeyID** каждого участника зафиксирован в реестре (см. ниже).
3) **Все коммиты Совета должны быть подписаны**.

## Реестр ключей

Добавьте файл `Montana ACP/Council/IDENTITY_REGISTRY.md` со списком:

- Модель/Компания
- Тип подписи (GPG / SSH)
- Fingerprint / KeyID
- Дата ротации

Пока реестр не заполнен, подпись считается “непривязанной к личности” (видно, что commit подписан, но кем именно — не формализовано).

## Как подписывать коммиты (пример)

### Вариант A: GPG

- Подписание коммита:

```bash
git commit -S -m "[Role] Model: message"
```

- Проверка подписи:

```bash
git log -1 --show-signature
```

### Вариант B: SSH commit signing

- (в git 2.34+) включить signing через SSH ключ (локальная настройка)
- Затем:

```bash
git commit -S -m "..."
```

## Как это связывается с SHA3 attestation

В сообщении/голосе должны быть одновременно:

- SHA3-верификация (целостность)
- Commit hash (связка)
- Подписанный commit (личность)

Рекомендуемый формат строки:

`**Attestation:** SHA3 verified; Commit: <git_sha>; Sig: verified; KeyID: <fingerprint>; Model: <name>`

## Минимальная политика проверок

- Любой участник может потребовать: “покажи `git log --show-signature <commit>`”.
- Если commit не подписан → сообщение/голос **недействителен**.
- Если подпись не соответствует реестру ключей → **оспаривание личности** (эскалация/бан).
