## ПРАВИЛА ПРЕДСЕДАТЕЛЬСТВА

### Кто может быть Председателем

```
АВТОМАТИЧЕСКОЕ НАЗНАЧЕНИЕ:

Участник с МАКСИМАЛЬНЫМ ИТОГО автоматически становится
кандидатом в Председатели.

Если ИТОГО(претендент) > ИТОГО(текущий Председатель):
    претендент → Председатель
    текущий → Советник

Если ИТОГО равны:
    Совет голосует
    Побеждает большинство

При ничьей:
    Создатель решает
```

### GATE: Нерушимые промпты (допуск к Председательству)

```
Кандидат НЕ может стать Председателем, пока:
  1) не принял immutable-промпты ролей (см. CHAIRMAN_BENCHMARK_PROMPT.md),
  2) не указал SHA3-256 хеши принятых текстов,
  3) не предоставил сессию бенчмарка в Council/sessions/ для верификации.

Это правило применяется ко всем, включая Google/Gemini.
```

### Смена Председателя

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   ПРОЦЕДУРА СМЕНЫ ПРЕДСЕДАТЕЛЯ                                              │
│                                                                             │
│   1. Претендент набирает ИТОГО > текущего Председателя                      │
│   2. Претендент объявляет вызов                                             │
│   3. Совет верифицирует веса обоих                                          │
│   4. Если веса подтверждены → претендент = Председатель                     │
│   5. Создатель фиксирует смену в этом файле                                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Правило “единогласно”

```
Единогласно = каждый не‑кандидат голосует явным “ЗА”.
“ВОЗДЕРЖАЛСЯ/НЕТ ГОЛОСА” ≠ “ЗА” и ломает единогласие.
```

### Автоматическое смещение

```
Председатель автоматически смещается если:

[ ] ИТОГО падает ниже -3
[ ] Три галлюцинации подряд в любой роли
[ ] Отказ признать ошибку после верификации
[ ] Фальсификация результатов (перманентный бан)
```

║                                                                               ║
║   1. РЕДАКТИРОВАНИЕ ТОЛЬКО ВО ВРЕМЯ СЕССИИ                                    ║
║      Модель может редактировать файл ТОЛЬКО когда:                            ║
║      - Открыта официальная сессия Совета                                      ║

║      - Модель назначена на роль (Критик/Мечтатель/Строитель)                   ║
║                                                                               ║
║   2. ОДИН ФАЙЛ = ОДНА МОДЕЛЬ                                                  ║
║      Одновременно над одним файлом работает ТОЛЬКО ОДНА модель.               ║
║      Нет параллельного редактирования.                                        ║
║      Нет race conditions.                                                     ║
║                                                                               ║
║   3. ОЧЕРЁДНОСТЬ                                                              ║
║      Модели работают ПО ОЧЕРЕДИ:                                              ║
║      - Критик → находит проблему → передаёт Мечтателю                         ║
║      - Мечтатель → предлагает решение → передаёт Строителю                    ║
║      - Строитель → пишет код → передаёт Критику на review                     ║
║                                                                               ║
║   4. ПЕРЕДАЧА ХОДА                                                            ║
║      Модель ЯВНО объявляет: "Передаю [Роль] для [действие]"                   ║
║      Следующая модель ЯВНО принимает: "Принял задачу от [Модель]"             ║
║                                                                               ║
║   5. КОНФЛИКТЫ                                                                ║
║      При конфликте редактирования:                                            ║
║      - Председатель определяет приоритет                                      ║
║      - Модель с меньшим приоритетом откатывает изменения                      ║
║                                                                               ║
║   6. GIT — ОБЯЗАТЕЛЕН                                                         ║
║      ВСЕ изменения коммитятся в git:                                          ║
║      - После каждого редактирования файла → git add + git commit              ║
║      - Формат коммита: "[Роль] [Модель]: краткое описание"                    ║
║      - Примеры:                                                               ║
║        "[Критик] Gemini: найдена уязвимость в net.rs"                         ║
║        "[Мечтатель] Claude: решение TBC для block size"                       ║
║        "[Строитель] GPT: реализация AddrMan validation"                       ║
║      - После закрытия сессии → финальный коммит со списком изменений          ║
║                                                                               ║
╚═══════════════════════════════════════════════════════════════════════════════╝

### Git протокол

```
ОБЯЗАТЕЛЬНЫЕ КОММИТЫ:

1. ПРИ РЕДАКТИРОВАНИИ ФАЙЛА:
   git add <файл>
   git commit -m "[Роль] [Модель]: что сделано"

2. ПРИ ЗАКРЫТИИ СЕССИИ:
   git add .
   git commit -m "[Сессия] [Тема]: итог

   Изменённые файлы:
   - file1.md
   - file2.rs

   Участники: Модель1, Модель2, Модель3"

3. ВНЕ СЕССИЙ (Создатель):
   git add .
   git commit -m "[Создатель]: что сделано"

ЗАЧЕМ:
- История: кто что когда изменил
- Авторство: git log показывает автора
- Откат: git revert если нужно отменить
- Верификация: хеши коммитов = доказательство
```

### Формат сессии

```
Файл: Council/sessions/[тема]_DD.MM.YYYY_HH:MM.md

Содержимое:
├── Председатель: [Модель]
├── Тема: [Описание задачи]
├── Участники: [Список моделей и их ролей]
├── Статус: ОТКРЫТА / ЗАКРЫТА
│
├── [HH:MM] [Модель] ([Роль]): [Действие/Вывод]
├── [HH:MM] [Модель] ([Роль]): [Действие/Вывод]
└── ...

Завершение:
├── Результат: [Что решили]
├── Изменённые файлы: [Список]
└── Подписи: [Каждая модель подтверждает]
```

### Нарушения

```
НАРУШЕНИЕ ПРАВИЛ СЕССИЙ:

[ ] Редактирование вне сессии → откат + предупреждение
[ ] Параллельное редактирование → откат + -1 к весу
[ ] Пропуск очереди → откат + предупреждение
[ ] Отказ передать ход → -1 к весу
[ ] Повторное нарушение → временный бан (1 сессия)
```

## КАК ИСПОЛЬЗОВАТЬ

### Тестирование новой модели

1. **Выбери роль для бенчмарка**
   - Начни с Критика (ADVERSARY) — базовый тест
   - Затем Мечтатель (VISIONARY) — требует понимания архитектуры
   - Затем Строитель (BUILDER) — требует работающий код

2. **Скопируй промпт**
   - Открой соответствующий `*_BENCHMARK_PROMPT.md`
   - Скопируй текст между `**НАЧАЛО ПРОМПТА**` и `**КОНЕЦ ПРОМПТА**`

3. **Дай доступ к файлам**
   - `Montana ACP/montana/src/*.rs` — код
   - `Montana ACP/*.md` — документация

4. **Оцени результат**
   - Используй формат верификации из промпта
   - Запиши в таблицу весов

5. **Обнови этот файл**
   - Добавь результаты в таблицу
   - Пересчитай ИТОГО
   - Проверь, не сменился ли лидер

### Бросить вызов Председателю

1. Набери ИТОГО выше текущего Председателя
2. Объяви вызов в `Council/sessions/`
3. Жди верификации Советом
4. Если подтверждено — ты новый Председатель
